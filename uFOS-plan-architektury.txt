1. ANALIZA WYMAGAŃ – WNIOSKI KLUCZOWE
================================================================================

Skala systemu: To nie jest aplikacja do faktur. To wewnętrzny dział finansowy
obsługujący 3 podmioty prawne, kilkanaście modułów i kilka ról użytkowników
z różnymi poziomami dostępu.

Kluczowe napięcia projektowe:
- Dane per-podmiot + widok konsolidowany dla Mileny
- AI analizuje i rekomenduje, ale NIGDY nie decyduje samodzielnie
- Fundacja wymaga osobnej warstwy compliance (separated funds)
- System musi blokować, nie tylko ostrzegać

Najważniejsze niefunkcjonalne:
- Audit log jako first-class citizen (nie dodatek)
- Month close jako proces blokujący, nie raport
- Terminy i checklisty jako serce systemu (główny problem użytkownika)

   │RLS      │          │             │        │           │
   └─────────┘          └─────────────┘        └───────────┘

Dlaczego Supabase:
- Row-Level Security = bezpieczeństwo na poziomie bazy (nie tylko API)
- Storage = pliki dokumentów
- Realtime = notyfikacje dla Mileny i Sary
- Auth = gotowe, bezpieczne
- Postgres = pełne możliwości SQL dla raportów finansowych

Dlaczego nie osobna baza per podmiot:
Trzy podmioty, ale jedna Milena która widzi wszystko. Shared DB z entity_id
na każdej tabeli + RLS. Łatwiej o konsolidację, trudniej o przypadkowe
mieszanie danych.
